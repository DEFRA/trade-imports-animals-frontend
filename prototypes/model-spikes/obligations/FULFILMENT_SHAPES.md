# Fulfilment shapes by example

Walks through the flat, composite-key `fulfilments` map by watching it
grow (and shrink) as a user progresses through the spike's obligations.
Each state is a snapshot of the whole `fulfilments` map; unless a
section says otherwise, states build on each other.

## Terminology

- **Fulfilment** — what an obligation is fulfilled by. For a
  single-cardinality obligation it's one atomic value; for an indexed
  obligation it's a keyed collection of **records**.
- **Record** — one atomic filling-in inside an indexed fulfilment. A
  single-cardinality fulfilment is one record.
- **fulfilmentId** — identifier of one record. For single-cardinality,
  `fulfilmentId === obligationId`.

See obligations.md §Terminology for the full glossary.

## Reading these examples

- **Obligation ids** appear as `[<name>.id]` — computed-property syntax
  that resolves to a distinct UUID per obligation. The actual UUIDs live
  in `obligations.js`. For example, State A3 below expands to:

  ```js
  fulfilments = {
    'b2a1c3d4-e5f6-4a7b-8c9d-0e1f2a3b4c5e': true, // hasClaims.id
    'e8f7d6c5-b4a3-4291-8074-6c5b4a392817': { c1: 'accident' } // claimType.id
  }
  ```

  Every other example in this doc uses `[<name>.id]` shorthand for the
  same reason as the fulfilment-id mnemonics — UUIDs are noise for
  illustrative purposes.

- **User-driven fulfilment ids** (`d1`, `d2`, `c1`, `p1`, `a1` …) are
  short mnemonics for readability only. In practice they are opaque
  ids generated by the orchestrator when the user adds a record —
  typically ULIDs (e.g. `01H8XK7M5RW6QYJ2AB`) or UUIDs. The tests use
  ULIDs.
- **Derived fulfilment ids** are the controller's answer values, not
  opaque ids. In §E, `'turbo'` / `'alloys'` / `'suspension'` are the
  strings the user picked in the `modifications` multi-select;
  `modificationCost.applyTo` returns them as its fulfilment id array,
  so the composite key IS the mod name.
- **Seeded fulfilment ids** would be whatever the external system
  provides (policy reference numbers, DB ids, domain codes). Not used
  in this spike.

**Delimiter constraint.** Composite keys use `/` as the segment
delimiter. UUID/ULID characters don't include `/`, and the derived
controllers in this spike use simple word strings, so `/` is safe.
Any future service whose derived controller values could contain `/`
(free-text codes, path-shaped values) would need to reserve or escape.
As a rule: **fulfilment ids MUST NOT contain the path delimiter** —
validated at write time.

## A. Single-cardinality progression (`hasClaims` + `claim` group)

**A0** — nothing answered:

```js
fulfilments = {}
```

**A1** — hasClaims = No:

```js
fulfilments = { [hasClaims.id]: false }
```

**A2** — hasClaims = Yes; no claims yet:

```js
fulfilments = { [hasClaims.id]: true }
```

The `claim` group is now in scope (its `applyTo` sees hasClaims=true).
No claim instances exist yet.

**A3** — user adds claim `c1`, answers its type:

```js
fulfilments = {
  [hasClaims.id]: true,
  [claimType.id]: { c1: 'accident' }
}
```

No entry for `claim.id` itself — the group's fulfilment is inferred,
not stored. Presence of `c1` is inferred from `claimType`'s
composite-key prefix.

**A4** — user answers the amount too:

```js
fulfilments = {
  [hasClaims.id]: true,
  [claimType.id]: { c1: 'accident' },
  [claimAmount.id]: { c1: '1200' }
}
```

**A5** — user adds a second claim `c2`:

```js
fulfilments = {
  [hasClaims.id]: true,
  [claimType.id]: { c1: 'accident', c2: 'theft' },
  [claimAmount.id]: { c1: '1200', c2: '500' }
}
```

## B. Depth-1 indexed leaf (`driver` → `driverAddress`)

`driver` is a top-level group (always in scope). `driverAddress` is an
indexed leaf within it — composite key has 2 segments:
`driverId/addressId`.

**B1** — user adds driver `d1`, answers name:

```js
fulfilments = {
  [driverFullName.id]: { d1: 'Alex' }
}
```

**B2** — user adds an address for `d1`:

```js
fulfilments = {
  [driverFullName.id]: { d1: 'Alex' },
  [driverAddress.id]: { 'd1/a1': { line1: '10 High St' /* ... */ } }
}
```

**B3** — second address for `d1`:

```js
fulfilments = {
  [driverFullName.id]: { d1: 'Alex' },
  [driverAddress.id]: {
    'd1/a1': {
      /* ... */
    },
    'd1/a2': {
      /* ... */
    }
  }
}
```

**B4** — second driver `d2`, name only:

```js
fulfilments = {
  [driverFullName.id]: { d1: 'Alex', d2: 'Sam' },
  [driverAddress.id]: {
    'd1/a1': {
      /* ... */
    },
    'd1/a2': {
      /* ... */
    }
  }
}
```

`d2` has no addresses yet; no entry under `driverAddress.id` for `d2`.

## C. Depth-2 chain (`driver` → `driverClaim` → `driverClaimOtherParty`)

`driverClaim` is a structural group with no field records in the spike.
A claim's presence has to come from at least one party under it — the
"no empty group instances" convention.

**C1** — driver `d1`, name only:

```js
fulfilments = {
  [driverFullName.id]: { d1: 'Alex' }
}
```

**C2** — user starts adding a claim, answers first party `p1`:

```js
fulfilments = {
  [driverFullName.id]: { d1: 'Alex' },
  [driverClaimOtherParty.id]: {
    'd1/c1/p1': { name: 'Other Driver', role: 'other-driver' }
  }
}
```

3-segment key. Claim `d1/c1` is inferred from the prefix.

**C3** — second party in same claim:

```js
fulfilments = {
  [driverFullName.id]: { d1: 'Alex' },
  [driverClaimOtherParty.id]: {
    'd1/c1/p1': {
      /* ... */
    },
    'd1/c1/p2': {
      /* ... */
    }
  }
}
```

**C4** — second claim `c2` with one party:

```js
fulfilments = {
  [driverFullName.id]: { d1: 'Alex' },
  [driverClaimOtherParty.id]: {
    'd1/c1/p1': {
      /* ... */
    },
    'd1/c1/p2': {
      /* ... */
    },
    'd1/c2/p3': {
      /* ... */
    }
  }
}
```

Both `d1/c1` and `d1/c2` are inferred as `driverClaim` instances
under driver `d1`.

## D. Purge on gate flip (`hasClaims` → false)

Start from A5 (two claims with type + amount). User navigates back and
flips hasClaims to false.

**Before evaluate** (raw write from the form POST):

```js
fulfilments = {
  [hasClaims.id]: false,
  [claimType.id]: { c1: 'accident', c2: 'theft' },
  [claimAmount.id]: { c1: '1200', c2: '500' }
}
```

**After evaluate** (amended):

```js
fulfilments = {
  [hasClaims.id]: false
}
```

`claim.applyTo` returns `{ inScope: false }`. Every composite key whose
path visits `claim` — that's every key under `claimType` and
`claimAmount` — is dropped. If hasClaims flips back to true later, the
`fulfilments` map is genuinely empty (no rehydration, consistent with
`obligations.md` §Lifecycle for derived).

## E. Derived indexed leaf (`modificationCost`)

Ids come from the `modifications` multi-select controller. Composite
key length = 1 (no parent group; own-inner-id only).

**E1** — hasModifications = true, modifications = `['turbo', 'alloys']`:

```js
fulfilments = {
  [hasModifications.id]: true,
  [modifications.id]: ['turbo', 'alloys']
}
```

`modificationCost` is now in scope with record-id set
`{ 'turbo', 'alloys' }` (returned by its `applyTo`). No records
stored yet.

**E2** — user answers cost for `'turbo'`:

```js
fulfilments = {
  [hasModifications.id]: true,
  [modifications.id]: ['turbo', 'alloys'],
  [modificationCost.id]: { turbo: '800' }
}
```

**E3** — user removes `'alloys'`, adds `'suspension'`. Nothing to purge
(no cost was recorded for alloys):

```js
fulfilments = {
  [hasModifications.id]: true,
  [modifications.id]: ['turbo', 'suspension'],
  [modificationCost.id]: { turbo: '800' }
}
```

**E4** — user answers cost for `'suspension'`:

```js
fulfilments = {
  [hasModifications.id]: true,
  [modifications.id]: ['turbo', 'suspension'],
  [modificationCost.id]: { turbo: '800', suspension: '600' }
}
```

**E5** — user removes `'turbo'` from the multi-select. Before evaluate:

```js
fulfilments = {
  [hasModifications.id]: true,
  [modifications.id]: ['suspension'],
  [modificationCost.id]: { turbo: '800', suspension: '600' }
}
```

**After evaluate** (amended):

```js
fulfilments = {
  [hasModifications.id]: true,
  [modifications.id]: ['suspension'],
  [modificationCost.id]: { suspension: '600' }
}
```

`'turbo'` is purged because `modificationCost.applyTo`'s returned
record-id array no longer contains it.

## What falls out

- **Groups have no stored fulfilment.** Their instance ids are inferred
  from descendants' composite-key prefixes.
- **Composite key length = number of instance dimensions** from root
  down to the value being held.
- **A record implies presence of every prefix along its path.** Field
  records and indexed leaves are how the model learns "this claim
  exists" or "this driver's claim's other party exists".
- **Purge is one rule** — drop composite keys whose path visits an
  out-of-scope group, OR whose innermost segment is not in a derived
  leaf's `applyTo`-returned record-id set.
- **User-driven inner-level ids are self-valid** — being in the
  `fulfilments` map is enough; no external enumeration to check
  against.

## Not covered here (yet)

- **Seeded indexed leaves** — the third `indexedBy.source`. Follows
  the same shape as user-driven; the difference is that initial ids
  come from an orchestrator-fired seed handler at journey start.
- **Field records on nested groups** — the spike's `driverClaim` has
  no field records; a real service modelling per-claim `type` /
  `amount` / `date` would add them as `{ id, name, within: driverClaim,
status }` with records at composite key `driverId/claimId`.
- **Empty group instances** — a compound-record instance with zero
  filled fields. Not representable under the current model; if a
  concrete service needs it, the resolution is to add a designated
  hidden identity field on the group.
- **Concurrency / multi-tab writes** — persistence-layer concern,
  outside the scope of the model.
